## JS-verify-1 — 승인 기다림 없이 바로 카운트, 가짜만 걸러진다 (← P2-1 auto-verify)

**When** 운동을 막 끝내고 인증 사진을 올렸을 때,
**I want to** 누군가의 승인을 기다리지 않고 바로 내 인증이 카운트에 잡히고, 명백한 가짜만 걸러지길,
**so I can** 친구를 믿는 분위기에서 매끄럽게 인증하면서도 부정은 따로 걸러진다는 안심을 얻는다.

Acceptance (사용자 관측):

1. 사진을 올리면 별도 승인 단계 없이 바로 "오늘 인증함"으로 잡힌다(하루 1회 기준).
2. 대부분의 인증은 그대로 통과한다 — 기본이 통과다(친구 신뢰).
3. 누가 봐도 명백한 가짜(같은 사진 재탕·딴 데서 캡처 등)만 걸러져 카운트에서 빠진다. 단 피드에는 남는다.
4. 기계가 애매해서 확신 못 하는 건 드물게 따로 사람이 확인한다 — 무턱대고 막지 않는다.
5. 판정 결과를 바로 확인할 수 있고, 걸러졌다면 왜인지 알 수 있다.

## JS-verify-2 — 잘못 올린 사진을 마감 전 한 번 바로잡는다 (← P2-1 auto-verify, 사진 교체)

**When** 인증 사진을 잘못 올린 걸 마감 전에 알아챘을 때,
**I want to** 마감 전 딱 한 번 사진을 바로잡고,
**so I can** 단순 실수로 억울하게 인증을 날리지 않는다.

Acceptance:

1. 챌린지 마감 전이라면 사진을 1회 교체할 수 있다.
2. 교체하면 부정 여부를 다시 검사한다(교체로 검증을 우회 못 함).
3. 교체는 1회까지만 — 무한 교체로 악용할 수 없다.
4. 마감 후에는 교체할 수 없다.

## JS-verify-3 — 재탕·캡처 사진이 통하지 않는다는 공정함 (← P2-3 cheat-detect, 핵심)

**When** 친구들과 벌금을 걸고 챌린지를 할 때,
**I want to** 남이 예전 사진을 재탕하거나 딴 데서 캡처한 사진으로 인증을 통과시키지 못한다고 믿고,
**so I can** 나만 정직하게 운동하고 손해 보는 게 아니라는 공정함을 느낀다.

Acceptance:

1. 이미 한 번 쓴 사진(나든 누구든)을 다시 올리면 인증으로 인정되지 않는다.
2. 운동한 시점과 동떨어진 시각에 찍힌 사진은 의심으로 걸린다.
3. 화면 캡처(스크린샷)는 인증으로 인정되지 않는다.
4. 걸러진 인증은 카운트에서 빠지되 피드엔 남아, 그룹이 무슨 일이 있었는지 안다.

## JS-verify-4 — 올리기 전에 안 될 사진은 미리 알려준다 (← P2-3 cheat-detect 온디바이스 사전검증)

**When** 인증 사진을 고르거나 찍어서 올리려 할 때,
**I want to** 흐리거나 스크린샷 같은 명백히 안 될 사진은 올리기 전에 미리 알려주길,
**so I can** 헛되이 올렸다 거절당하는 일 없이 바로 다시 찍는다.

Acceptance:

1. 명백히 부적합한 사진(심하게 흐림·스크린샷)은 올리기 전 단계에서 안내가 뜬다.
2. 안내는 막아서는 게 아니라 "다시 찍기"를 권하는 수준이다(최종 판정은 제출 후).
3. 사전 안내는 빠르게 뜬다 — 업로드 완료를 기다리지 않는다.

## JS-verify-5 — 수상한 인증을 그룹이 함께, 익명으로 거른다 (← P2-2 peer-reject)

**When** 친구의 인증이 보기에 명백히 수상할 때,
**I want to** 그룹장 혼자가 아니라 우리 그룹이 함께, 그것도 익명으로 그 인증을 걸러내고,
**so I can** 누구 하나 미워지는 일 없이 공정하게 부정을 거른다.

Acceptance:

1. 인증 카드에 🟨 반려(옐로카드)를 한 번 탭으로 표시할 수 있다 — 응원(Kudos)과는 별개다.
2. 누가 눌렀는지는 숨겨지고, 몇 명이 눌렀는지만 익명으로 집계된다.
3. 본인을 뺀 참가자의 과반이 반려하면 그 인증은 카운트에서 빠진다. 자기 인증은 자기가 반려할 수 없다.
4. 마음을 바꿔 다시 누르면(토글) 과반 아래로 떨어질 때 인증이 복원된다. 반려는 정산 전(마감 후 일정 기간)까지 유효하다.
5. 이 그룹 다수결이 "그룹장 단독 판정"을 대신한다.

## JS-verify-6 — 그룹장은 일일이 확인 안 해도, 혼자 판정 안 해도 된다 (← P2-4 owner-load)

**When** 그룹장으로서 챌린지를 운영할 때,
**I want to** 인증마다 일일이 확인하지 않아도 되고 부정 판단을 나 혼자 짊어지지도 않길,
**so I can** 운영 부담과 친구 사이 껄끄러움 없이 챌린지에 집중한다.

Acceptance:

1. 기본이 자동 통과라 그룹장이 인증을 하나하나 승인할 필요가 없다.
2. 반려는 그룹 다수결로 정해지고, 그룹장도 똑같이 한 표만 행사한다(특별 권한 없음).
3. 걸러진 인증·반려가 비정상적으로 많아지면 그룹에 알림이 와서 상황(오작동·갈등)을 알 수 있다.

## Edge Cases (raw — normalize 시 risks-assumptions로도 흡수)

- 검증 서비스 일시 다운 → 막지 말고 "나중에 사람이 확인"으로 부드럽게 넘긴다(graceful).
- 판정 비용 급증(사진 폭증) → 올리기 전 1차 거름(JS-verify-4) + 볼륨 모니터.
- 오프라인에서 제출 → 큐에 담아뒀다 연결되면 동기화.
- 기계 오판으로 정상 인증이 걸림 → 사진 1회 교체(JS-verify-2)로 정정 / 피어 복원으로 구제.
- AI 생성·재촬영·메타 조작으로 우회 → MVP 범위 밖, Fast-follow 하드닝(red-team).
- **G1(부정탐지 임계 θ) 미확정** → cheat-detect·auto-verify 관련 Agent Task는 `blocked`(θ 주입 후 활성, DECISION_NEEDED `G1-θ`).
