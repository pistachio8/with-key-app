# 코딩 스타일

> Karpathy 4원칙(생각·단순함·외과적 수정·목표 중심) + 불변성. 공통 품질 기준은 [`../../../docs/QUALITY_GATE.md`](../../../docs/QUALITY_GATE.md)를 우선합니다.

## 1. 코드 작성 전에 생각하기

추측하지 말고, 혼란을 숨기지 말고, 선택지를 드러낸다.

- 가정을 명확히. 불확실하면 질문.
- 여러 해석이 가능하면 모두 제시 — 조용히 하나를 선택하지 않는다.
- 더 단순한 접근이 있다면 언급.
- 명확하지 않으면 멈추고, 무엇이 헷갈리는지 구체적으로 말한다.

## 2. 단순함 우선

문제를 해결하는 최소한의 코드. 불필요한 것 금지.

- 요청받지 않은 기능·추상화·"유연성"·발생할 수 없는 에러 처리 추가 금지.
- 200줄로 작성했는데 50줄로 가능하면 다시 작성.

기준: "시니어가 이걸 보고 과하게 복잡하다고 할까?" → 그렇다면 단순화.

## 3. 외과적 수정

필요한 부분만 건드린다. 정리는 본인이 만든 것만.

- 주변 코드/주석/포맷을 "개선"하지 않는다.
- 문제 없는 코드 리팩토링 금지. 기존 스타일을 따른다.
- 관련 없는 죽은 코드는 언급만, 삭제 금지.
- 본인 변경으로 사용되지 않게 된 import/변수만 제거.

검증: 변경된 모든 라인은 사용자 요청과 직접 연결되어야 한다.

## 4. 목표 중심 실행

성공 기준을 정의하고, 검증될 때까지 반복.

- "validation 추가" → "잘못된 입력 테스트 작성 → 통과시키기"
- "버그 수정" → "버그 재현 테스트 작성 → 통과시키기"
- "X 리팩토링" → "변경 전/후 테스트 모두 통과 확인"

여러 단계면 `1. [단계] → 검증: [확인 방법]` 형식으로 짧은 계획 작성.

## 불변성

기존 객체를 변경하지 말고 새 객체를 만든다. 사이드 이펙트 방지·디버깅 용이·동시성 안전. (구체 패턴은 [`../typescript/patterns.md`](../typescript/patterns.md))

## 파일 구성·에러·입력 검증

- 작은 파일 다수 > 큰 파일 소수. 일반적 200~400줄, 최대 800줄. 기능/도메인별 분리.
- 에러는 모든 레벨에서 명시적으로. UI는 사용자 친화 메시지, 서버는 컨텍스트 로깅. 조용히 무시 금지.
- 입력은 시스템 경계에서 검증. 가능하면 schema 기반(zod). 명확한 메시지로 빠르게 실패. 외부 데이터(API/사용자/파일) 신뢰 금지.
