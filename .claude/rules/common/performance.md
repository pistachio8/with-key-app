## 컨텍스트 윈도우 관리

컨텍스트는 토큰 수 고정값보다 작업 단위와 현재 사용률 기준으로 관리한다.
200k 같은 단일 임계값은 모델별 컨텍스트 크기와 맞지 않을 수 있으므로 단독 기준으로 쓰지 않는다.

컨텍스트 윈도우의 마지막 20%에서는 다음을 피하세요:
- 대규모 리팩토링
- 여러 파일에 걸친 기능 구현
- 복잡한 상호작용 디버깅

다음 시점에는 `/compact`를 수행하거나 사용자에게 제안한다:

- 작업 배치가 끝났을 때
- 주제가 바뀌기 전
- 긴 로그, 긴 문서, 대량 검색 결과에서 필요한 사실을 정리한 뒤
- 이미지 업로드에서 필요한 관찰 사실과 결정 사항을 텍스트로 남긴 뒤
- `/context`가 약 70-80% 이상일 때

이미지 원본의 세부 시각 정보가 아직 판단 근거라면 즉시 압축하지 않는다.

컨텍스트 민감도가 낮은 작업:
- 단일 파일 수정
- 독립적인 유틸리티 생성
- 문서 업데이트
- 단순한 버그 수정

## 탐색 비용 절감

- 파일 탐색은 Glob/Grep/`rg`를 먼저 사용한다.
- 큰 파일은 전체를 읽기 전에 관련 symbol, 경로, 테스트 이름을 먼저 좁힌다.
- 긴 출력은 필요한 줄 주변만 다시 읽고, 검색 결과 전체를 컨텍스트에 붙이지 않는다.
- shell의 `grep/find/cat` 전체 덤프보다 Claude Code의 Grep/Glob/Read 또는 `rg` 기반 범위 탐색을 우선한다.

## Prompt caching

- 지원 여부가 확인된 환경 변수만 설정한다.
- 현재 근거가 확인되지 않은 `ANTHROPIC_CACHE_TTL`은 프로젝트 규칙에 추가하지 않는다.
- 5분 TTL 강제는 1시간 캐시 이점을 줄일 수 있으므로 비용 절감 기본값으로 두지 않는다.

## 확장 사고 + 계획 모드

확장 사고는 기본적으로 활성화되어 있으며, 내부 추론을 위해 최대 31,999 토큰을 예약합니다.

확장 사고 제어 방법:
- **전환**: Option+T (macOS) / Alt+T (Windows/Linux)
- **설정**: `~/.claude/settings.json`에서 `alwaysThinkingEnabled` 설정
- **예산 제한**: `export MAX_THINKING_TOKENS=10000`
- **상세 모드**: Ctrl+O로 사고 출력 확인

깊은 추론이 필요한 복잡한 작업:
1. 확장 사고가 활성화되어 있는지 확인 (기본 활성)
2. 구조적 접근을 위해 **계획 모드** 활성화
3. 철저한 분석을 위해 여러 라운드의 비판 수행
4. 다양한 관점을 위해 역할 분리 서브에이전트 사용

## 빌드 문제 해결

빌드 실패 시:
1. **everything-claude-code:build-error-resolver** 에이전트 사용
2. 에러 메시지 분석
3. 점진적으로 수정
4. 각 수정 후 검증
