# ADR-0043: 적립 포인트 환불·중도 탈퇴자 보증금 처리 정책 (Q2)

**Date**: 2026-06-29
**Status**: accepted
**Deciders**: pistachio8 (PO 수락 2026-06-29)
**관련**: PRD §7 Q2([01-rn-mvp-prd.md](../migration/01-rn-mvp-prd.md) §7 — 적립 포인트 환불·탈퇴자 보증금 처리, Owner 법무+PO) · RA-Vi2([risks-assumptions.md](../pm/risks-assumptions.md) — 탈퇴자·환불 정책 분쟁 없는 운영) · 게이트 **G2**(법무 검토) · ADR-0032(settlement-verification-data-model) · `AC-settle-1·2·3`·`AC-points-use-1`

## Context

게이트 G2(법무 검토 1회)는 "ⓑ적립/번들 포인트(현금 충전 아님)로 시작 시 환불 의무·사행성·선불전자지급수단 리스크 ≈0"을 확인해야 열린다(RA-Vi1). 그런데 G2의 실질 병목은 규제 3축이 아니라 **Q2 — 적립 포인트의 환불·중도 탈퇴자 보증금 처리 정책이 비어 있어 법무가 사인할 근거가 부족**하다는 점이다. 규제 3축(선불전자지급수단·사행성·환불)은 이미 설계 결정으로 리스크를 ≈0로 눌렀다(현금 충전 배제·공동 주머니 이월·closed-loop). 남은 건 운영 정책 한 칸이다.

PRD §7은 Q2를 Open Question으로, RA-Vi2는 confidence Low(미해결)로 두고, mitigation을 "Q2 확정 전 경로 비활성, 정책 고지"로 잡아 두었다. P1 엣지 케이스에도 "중도 탈퇴자 보증금 처리(정책, Open Q)"·"환불 요청(ⓑ적립이라 원칙적 비현금, 정책 고지)"이 미정으로 남아 있다.

코드·결정 사실 네 가지가 이 정책을 강하게 제약한다.

1. **포인트는 closed-loop 무상 적립이다.** ⓑ적립/번들 포인트는 현금 충전이 아니라 적립·구독 번들로 지급되고(현금화 불가), `AC-points-use-1`이 이를 불변식으로 강제한다. 현금 충전 포인트는 G2 통과 후 Fast-follow로 분리돼 MVP 범위 밖이다.
2. **정산 모델은 공동 주머니 방식이다.** 미달분(forfeit)은 그룹 공동 주머니로 귀속돼 다음 챌린지 보증금으로 이월된다(`AC-settle-1`). 개인 간 재분배는 도박 위험으로 미채택(`AC-settle-6`). 공동 주머니는 "그룹이 살아있는 한 무기한 보관, 그룹 삭제 시에만 소멸"(`AC-settle-2`, Q3 확정).
3. **원장에 `refund` reason이 이미 있다.** `point_ledger.reason` CHECK 제약에 `bundle_grant`·`deposit_hold`·`deposit_release`·`penalty`·`distribution`·`refund`·`penalty_debt_carryover`가 정의돼 있다([BE_SCHEMA.md:333](../BE_SCHEMA.md)). 환불 이동 타입 자체는 원장이 표현할 수 있다 — 비어 있는 건 "언제·얼마를 환불하는가"의 정책이다.
4. **멤버 self 탈퇴와 계정 삭제 경로가 별개로 존재한다.** `group_members`는 self 탈퇴를 RLS로 허용하고([BE_SCHEMA.md:573](../BE_SCHEMA.md)), 계정 삭제는 `user_deletion_queue` + 30일 지연 삭제 크론으로 처리된다([BE_SCHEMA.md:789](../BE_SCHEMA.md)). 둘 다 hold된 보증금이 걸려 있을 때의 처리가 미정이다.

이 정책은 G2 법무 검토에 제출할 입력이자, 정산 데이터 모델(보증금 환급 vs 몰수)의 방향을 정하는 되돌리기 비용 큰 결정이라 ADR 대상이다.

## Decision

**적립 포인트는 무상 closed-loop 자산이므로 현금 환불 의무를 지지 않고, 챌린지 진행 중 중도 탈퇴자의 hold된 보증금은 전액 forfeit하여 그룹 공동 주머니로 귀속한다.** 세부:

- **DP1 — 중도 탈퇴자 hold = 전액 forfeit → 공동 주머니.** 챌린지 진행 중 탈퇴는 미완주(=미달)로 간주해 hold된 적립분 전액을 그룹 공동 주머니로 귀속한다(원금 환급 없음). **왜**: 탈퇴에 원금 환급을 붙이면 손실회피 압박이 무력화되고 탈퇴 인센티브가 생긴다 — `AC-settle-1`의 "미달분 forfeit → 공동 주머니" 흐름과 동일선상. 서약 동의 시점에 "탈퇴 시 보증금 미반환"을 명시 고지하는 것을 전제로 한다(DP4).
  - **forfeit 실현 메커니즘은 기존 정산(미달 forfeit) 재사용을 원칙으로 하되, 정확한 원장 reason·시점(탈퇴 즉시 forfeit 행 기록 vs 정산 시 미달자로 일괄 처리)은 구현 시 확정한다.** 새 reason 신설은 최소화한다(`distribution`/`penalty` 재사용 우선 검토). **왜**: ADR-0032 정산 불변성·doneCount 재계산 흐름과의 정합은 로컬 Supabase 실호출로 검증해야 확정 가능하다.
- **DP2 — 환불 요청 = 현금 환급 의무 없음, closed-loop 잔액 유지.** ⓑ적립/번들 포인트는 무상 지급이라 전자상거래법상 청약철회·환불 대상이 아니다. 사용자의 "돌려줘" 요청에 현금을 지급하지 않으며, 잔액은 앱 내 closed-loop로만 유지·소진한다. **왜**: 현금화 경로를 여는 순간 환불 의무·자금이체·선불전자지급수단 규제가 발생한다(`AC-settle-3`). 현금 충전 포인트가 도입되는 Fast-follow에서는 충전분에 한해 환불 정책을 별도로 다룬다(본 ADR 범위 밖).
- **DP3 — 계정 삭제 시 개인 잔액·hold = 소멸.** 계정 삭제(`user_deletion_queue`) 시 개인 적립 잔액은 현금화 불가라 청구권이 없으므로 소멸한다. 삭제 시점에 진행 중인 챌린지의 hold는 DP1(forfeit)을 적용한다. **왜**: 무상 적립 자산에 현금 청구권을 인정하지 않는다는 closed-loop 원칙의 귀결.
- **DP4 — 정책 고지 위치 = 서약 동의 + 약관.** "탈퇴 시 보증금(적립 포인트) 미반환"·"적립 포인트는 현금 환급 불가"를 서약 동의 화면과 이용약관에 명시한다. **왜**: RA-Vi2 mitigation("정책 고지")의 이행 — 사전 고지가 분쟁 방어선이자 법무 사인오프의 전제다. 약관 문구 확정·반영은 Fast-follow(G2 통과 후 노출 전).

이 ADR은 **Q2를 해소해 G2 법무 검토의 입력을 채우는 것**이지, G2 통과(법무 사인오프) 자체가 아니다. 규제 3축의 최종 법적 판단은 여전히 법무의 외부 결정으로 남는다.

## Alternatives Considered

### 1. DP1 대안 — 경과분만 forfeit, 잔여 release

- **Pros**: 탈퇴 시점까지 누적된 `confirmedPenalty`만 몰수하고 나머지 hold를 환급해 "낸 만큼만 잃는다"는 직관에 부합. 사용자 체감 공정성이 높다.
- **Cons**: 탈퇴 시점 기준 penalty 재계산 + 부분 release 로직이 추가되고, 정산 시 미달자 처리와 별개 경로가 생겨 POC 복잡도가 오른다.
- **Why not**: 손실회피 압박을 약화시키고(중도 이탈 비용 ↓) 구현·검증 부담만 늘린다. 친구 그룹 POC에선 전액 forfeit + 명확한 사전 고지가 더 단순하고 분쟁이 적다.

### 2. DP1 대안 — 전액 release(환급)

- **Pros**: 가장 단순하고 사용자 불만이 적다.
- **Cons**: 탈퇴하면 보증금을 100% 돌려받으므로 손실회피 압박(P1 핵심 가치)이 무력화되고, 미달 직전 탈퇴로 forfeit를 회피하는 우회로가 생긴다. 기존 "미달=forfeit" 설계와 정면 충돌.
- **Why not**: P1의 가치 가설(RA-V1, 손실회피로 완주율 ↑) 자체를 깨뜨린다.

### 3. DP2 대안 — 현금 환불 허용

- **Pros**: "보증금을 돌려준다"는 직관에 부합.
- **Cons**: 포인트→현금 전환은 선불전자지급수단·전자금융업·자금이체 규제 영역. PRD가 이미 현금 경로를 G2 통과 후 Fast-follow로 게이팅했다.
- **Why not**: closed-loop·현금화 차단(`AC-settle-3`)이 규제 3축을 ≈0로 누른 핵심 mitigation이다. 현금 환불을 허용하면 G2 가정 자체가 무너진다.

## Consequences

### 긍정적

- G2 법무 검토의 마지막 빈칸(Q2)이 채워져 게이트 통과의 입력이 완성된다 — RA-Vi2를 confidence Low(미해결)에서 정책 확정으로 전진.
- closed-loop·공동 주머니·손실회피라는 기존 설계 원칙에서 일관되게 도출돼 새 메커니즘을 거의 추가하지 않는다(원장 `refund`·`distribution` 재사용).
- 사전 고지(DP4)로 탈퇴·환불 분쟁의 1차 방어선을 확보한다.

### 부정적 / 비용

- 전액 forfeit(DP1)는 중도 탈퇴자에게 강한 정책이라, 사전 고지가 불충분하면 신뢰·CS 분쟁 리스크가 있다(DP4 고지 품질에 의존).
- 현금 충전 포인트가 도입되는 Fast-follow에서는 충전분 환불 정책을 본 ADR과 별개로 새로 정해야 한다(충전분은 현금성 → 환불 대상 가능).
- 탈퇴 forfeit의 정확한 원장 reason·시점이 본 ADR에서 확정되지 않아, 구현 task가 ADR-0032 정산 흐름과의 정합을 실호출로 검증해야 한다.

### 후속 영향

- PRD §7 Q2를 🟡 PO 정책 확정(본 ADR 링크, 법무 사인오프 잔여)으로, RA-Vi2를 confidence Medium(정책 확정)으로 갱신한다.
- 약관·서약 동의 화면 고지 문구(DP4) 작성·반영 — Fast-follow(G2 통과 후 사용자향 노출 전). 약관 원문은 법무 검토 대상.
- 탈퇴 forfeit 실현이 새 원장 reason을 요구하면 `point_ledger.reason` CHECK 확장 migration(append-only next available 번호) + `SettlementReason` union 갱신. 기존 reason 재사용으로 끝나면 migration 불필요.
- 이 ADR 자체가 G2 법무 검토 패키지의 구성 요소(환불·탈퇴 정책 파트)다. 규제 3축 논거(legal brief)는 별도 산출물로 묶어 함께 제출.
- 정책이 닿는 사용자향 노출(보증금 게이지·정산 화면·포인트 잔액)은 EVAL-0007·0008·0009 — 모두 G2 blocked, 본 ADR + 법무 사인오프 후 활성.
