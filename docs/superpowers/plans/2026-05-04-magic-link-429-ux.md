# Magic Link 429 UX 분기 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** JOURNAL 2026-05-01 "Web Push plan 완주" 의 다음 부채 중 **"`requestMagicLink` 429 UX 분기"** 해소. 현재 Supabase 의 이메일 전송 rate limit(60s/email · 시간당 쿼터) 에 걸리면 `upstream_error` 로 뭉뚱그려져 "잠시 후 다시 시도해 주세요" 만 뜬다. 429 를 별도 `rate_limited` ErrorCode 로 분리하고, 사용자에게 대기 안내 카피를 제공.

**Architecture:**
- 기존 `ErrorCode` 유니온에 **신규 `rate_limited`** 추가. Supabase AuthError 의 `status === 429` 또는 `code === "over_email_send_rate_limit"` 계열을 감지해 매핑.
- `error-messages.ts` default 에 `rate_limited` copy 추가 ("요청이 너무 많아요. 잠시 뒤 다시 시도해 주세요."). Login 페이지는 default copy 를 그대로 사용(별도 override 불필요).
- 다른 Server Action 은 건드리지 않음. rate limit 은 지금 시점에 auth OTP 외에는 의미 없는 분기.
- 기존 `mapSupabaseError` (Postgres 코드 매핑) 와는 별개 경로 — auth 는 AuthError 객체가 다르다. `_actions.ts` 에서 인라인 감지가 가장 단순.

**Non-Goals:**
- 남은 대기 시간(Retry-After) 실시간 카운트다운 — Supabase 는 헤더를 JS SDK 로 노출하지 않음. 정적 카피만.
- 전역 rate limit 처리 유틸 — auth OTP 하나만 필요 (YAGNI).
- 신규 이벤트(analytics) — `notification_sent` 같은 전용 이벤트 불필요.

## Self-Review

- ErrorCode 유니온 확장은 기존 5 코드 + 1 신규. `error-messages.ts` · `supabase-error.ts` · 각 action 에서 switch/map 의 exhaustiveness 는 `Record<ErrorCode, string>` 이 컴파일러가 강제.
- 429 감지 조건 2개(status numeric, code string prefix). Supabase JS SDK 의 `AuthApiError` 는 `status: number` 를 채우므로 primary 경로는 status. code prefix 는 보조 방어.
- 영향 범위: login 페이지 하나. 다른 Server Action 의 `upstream_error` 경로는 그대로 유지.
