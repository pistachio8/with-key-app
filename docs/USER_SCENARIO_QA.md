## 한눈에 보는 전체 흐름

처음 이 서비스를 쓰는 사람의 시간 순서입니다.

```
[로그인]  → 이메일로 매직링크 받기 → 링크 클릭 → 자동 로그인
   ↓
[그룹 만들기]  → "새 그룹 만들기" → 이름 입력 → (선택) 정산 계좌 등록
   ↓
[챌린지(서약서) 만들기]  → 제목/주 목표 횟수/기간/벌금 입력
   ↓
[친구 초대]  → 초대 링크 공유 → 친구가 링크 열어서 "참여하기"
   ↓
[모두 서명]  → 멤버 전원이 서약서에 동의 체크 → 챌린지 자동 시작
   ↓
[인증]  → "인증" 탭에서 사진/키워드 선택 → AI 일기 자동 생성 → 인증 완료
   ↓
[피드/응원]  → "피드"에서 멤버 인증을 보고 🔥💪👏 응원
   ↓
[종료]  → 기간 끝나면 → 회고(Recap) 화면에 결과 표시 (벌금은 표시만)
```

이 앱은 **PWA(Progressive Web App)** 입니다. 휴대폰 홈 화면에 설치하면 일반 앱처럼 보이고, 첫 화면이 `/home`으로 바로 열립니다. 브랜드 이름은 매니페스트 기준 **"from. with — 친구와 함께하는 운동 서약서"** (단축명 `FROMWITH`).

## 1. 화면 이동 — 어떤 버튼이 어떤 화면으로 보내나?

### Q1. 어떤 화면들이 있나요?

크게 두 묶음입니다.

**로그인 전 (또는 초대 받은 사람)**
- 로그인 페이지 (`/login`)
- 초대 링크 페이지 (`/invite/<토큰>`)

**로그인 후 메인 앱**
- 홈 (`/home`) — 내 그룹과 진행 중인 챌린지 요약
- 인증 (`/action`) — 운동 인증 작성
- 서약서 (`/pledge`) — 서약서 서명
- 피드 (`/feed`) — 멤버들의 인증 모음
- 회고 (`/recap`) — 끝난 챌린지의 결과
- 설정 (`/settings`) — 푸시 알림 설정
- 새 그룹 (`/group/new`)
- 새 서약서 (`/challenge/new?groupId=<id>`)
- 챌린지 상세 (`/challenge/<id>`)

화면 아래 항상 보이는 "탭바"에는 **홈 / 인증 / 서약서** 3개만 있습니다. 피드·회고·설정은 다른 곳에서 링크로 진입합니다.

> 📌 **그룹 상세 페이지는 따로 없습니다**. `/group/<id>`로 들어가도 페이지가 렌더되지 않습니다 (코드상 `page.tsx` 부재). 그룹의 상태는 챌린지 상세 페이지(`/challenge/<id>`)가 대신 보여줍니다 — 친구 초대 버튼·계좌 정보·멤버 진행률·피드가 모두 거기에 모여 있습니다.

- 탭바: `src/components/app-shell/bottom-nav.tsx:8-12`
- 그룹 상세 부재: `ls src/app/(app)/group/[id]/` → `_actions.ts`, `_components/`만 있고 `page.tsx` 없음

### Q2. 로그인은 어떻게 처리되나요?

이메일로 "매직링크"를 받아서 클릭하는 방식입니다. 비밀번호가 없습니다.

1. 로그인 화면에서 이메일 입력 → "이메일로 로그인 링크 받기"
2. 메일함에서 링크 클릭 → `/auth/callback?code=...`로 들어옴
3. 서버가 코드를 세션 쿠키로 바꾼 뒤 자동으로 `/home`으로 이동
4. 인증에 실패하면 `/login?error=auth`로 돌려보냄

이메일 OTP는 **이메일당 60초 쿨다운**이 있어, 60초 안에 같은 이메일로 다시 요청하면 `rate_limited` 에러가 뜨고 토스트는 "요청이 너무 많아요. 잠시 뒤 다시 시도해 주세요."가 됩니다.

카카오 로그인 버튼이 있지만 현재는 **비활성화**돼 있습니다 (v1 예정 — `disabled` 속성).

개발 환경 전용으로 `/auth/dev-login?token_hash=...` 라우트가 있습니다. SMTP 한도 문제 회피용이고 프로덕션에서는 404로 차단됩니다.

### Q3. 비로그인 상태에서 보호된 페이지로 가면 어떻게 되나요?

미들웨어가 가로챕니다. 로그인되지 않은 상태에서 `/login`·`/invite`·`/auth/*` 외의 경로에 접근하면 자동으로 `/login`으로 리다이렉트됩니다.

초대 링크는 다릅니다. `/invite/<토큰>`에 비로그인으로 들어오면 `/login?next=/invite/<토큰>` 형태로 이동했다가, 로그인 직후 다시 원래 초대 링크로 돌려보냅니다. 친구가 초대받았는데 로그인 때문에 길을 잃지 않게 한 장치입니다.

- 초대 링크의 next 파라미터: `src/app/(auth)/invite/[token]/page.tsx:17-20`
</details>

### Q5. 홈 화면은 그룹 개수에 따라 어떻게 다른가요?

**그룹이 0개**일 때: "아직 속한 그룹이 없어요" 빈 카드 + "새 그룹 만들기" 버튼만 보입니다.

**그룹이 1개**일 때: 그 그룹 카드가 크게 하나 보입니다.

**그룹이 2개 이상**일 때: 가로로 스와이프할 수 있는 카드 스트립이 됩니다 (각 카드 폭 화면의 85%).

각 그룹 카드의 내용은 그 그룹의 **가장 최근 챌린지(pending/accepted/active 중 하나)** 에 따라 다릅니다.

| 챌린지 상태 | 카드 내용 |
|------------|----------|
| 챌린지가 아예 없음 (또는 closed만) | "진행 중인 서약서가 없어요" + "새로운 서약서 만들기" 버튼 |
| pending/accepted/active | 진행률 카드 (제목 / D-N / 내 인증 횟수 / 모인 예정 벌금) + "현황 보기" 버튼 → `/challenge/<id>` |

**D-N 표시**: 종료까지 남은 일수입니다. `D-3`이면 3일 남음. 종료일에 도달하면 `D-0`이 되고, 카드에 "주간 정산 보기 →" 링크가 추가로 노출됩니다. 종료 시각이 지나면 코드상 `Math.max(0, ...)`로 처리되어 음수가 안 나옵니다.

### Q6. 홈에 "운동 시작" 단축 버튼은 언제 뜨나요?

내가 참여 중인 **진행 중 챌린지(active)가 정확히 1개**일 때만 뜹니다. 0개거나 2개 이상이면 "어떤 챌린지를 시작한다는 건가?" 모호함을 피하려고 보여주지 않습니다 (코드 주석: "모호함 방지").

또 **푸시 알림 구독을 안 한 사람**이 이 버튼을 누르면, 발송하지 않고 바로 안내 토스트만 뜹니다:

> 푸시 알림을 켜주세요  
> 그룹원에게 시작 알림을 보내려면 알림 설정이 필요해요.  
> [설정 열기]

"설정 열기" 액션 버튼을 누르면 `/settings`로 이동합니다.

### Q7. 회고(Recap) 화면은 어떻게 들어가나요?

탭바에 직접 가는 길이 없습니다. 두 가지 진입점이 있습니다.

1. 홈의 진행률 카드에서 D-0이 됐을 때 "주간 정산 보기 →" 링크가 추가로 표시됨
2. URL을 직접 입력 (예: 푸시 알림이 D-0 이후 알림에서 회고로 보내는 등 향후 확장)

들어가면 자동으로 "내가 참여했고 이미 종료 시각이 지난 챌린지" 중 가장 최근 한 개를 골라 결과를 표시합니다. 챌린지 status는 `active` 또는 `closed` 둘 다 처리합니다 — 시간이 지나도 자동 closed 전이가 코드상 없기 때문입니다.

종료된 챌린지가 없으면 "아직 끝난 챌린지가 없어요" 빈 화면이 보입니다.

### Q8. 피드 화면은 어느 챌린지의 인증을 보여주나요?

내가 참여한 챌린지 중에서 **가장 최근 active 챌린지 1개**의 인증만 모아서 보여줍니다. 활성 챌린지가 없으면 빈 화면 + 홈으로 가는 링크만 나옵니다.

피드 화면 우상단에는 챌린지 상세로 가는 "현황" 링크와, 미읽음 응원 개수가 있을 때만 표시되는 칩이 있습니다 (Q12 참고).

피드 화면에 진입하는 그 순간 `markFeedSeen` 함수가 자동으로 호출돼서 "마지막으로 피드를 본 시각"을 현재로 갱신합니다. 이게 미읽음 카운트의 기준선이 됩니다.

## 2. 팝업·모달 — 어떤 버튼이 무엇을 띄우나?

### Q9. 진짜 모달 창(Dialog)은 어디에 있나요?

코드 전체에서 **모달 창은 단 하나**입니다.

| 모달 | 어디서 뜨는가 | 안에 무엇이 있나 |
|------|-------------|-----------------|
| **정산 계좌 보기** | 챌린지 상세 화면의 "계좌 보기" 버튼 | 은행·예금주·**마스킹된** 계좌번호. "계좌번호 복사" 버튼을 누르면 평문이 클립보드로만 복사됨(화면에는 절대 표시 안 됨) |

모달의 **보안 설계**가 흥미롭습니다.

- 계좌번호 평문은 DB에 AES-256-GCM 으로 암호화돼 있고, 페이지 렌더링 시점에는 화면에도 클라이언트에도 내려오지 않습니다.
- "복사" 버튼을 누르면 그제야 서버 함수(`revealAccountNumber`)가 암호문을 복호화해서 평문을 응답으로 돌려주고, 클라이언트 코드가 그것을 곧장 `navigator.clipboard.writeText`로 클립보드에 넣습니다.
- 평문은 화면에 한 번도 그려지지 않고, 화면 캡처를 해도 마스킹된 계좌번호만 보입니다.

### Q10. 그 외 "버튼 누르면 뜨는 것"들은 모달이 아닌가요?

대부분 **토스트(Toast)** 입니다. 토스트는 화면 위쪽에 잠깐 떴다가 자동으로 사라지는 알림 메시지입니다 (`sonner` 라이브러리). 모달과 달리 사용자의 다른 행동을 막지 않습니다.

| 버튼 | 결과 | 형태 |
|------|------|------|
| 친구 초대 링크 공유 | 휴대폰 OS 공유 시트 (Web Share API 가능) 또는 클립보드 복사 + 성공 토스트 | OS 시트 또는 토스트 |
| 운동 시작 (정상) | 발송 결과에 따라 4가지 토스트 중 하나 (Q11 참고) | 토스트 |
| 인증하기 (성공) | "인증 완료!" 또는 "사진 없이 인증됐어요" 토스트 | 토스트 |
| 챌린지·그룹·서명 모든 액션 | 성공/실패 토스트 | 토스트 |
| 응원 이모지 (🔥💪👏) | 즉각 숫자 변경, 실패 시에만 토스트 | 인라인 |

**Web Share API와 폴백**:

- 브라우저가 Web Share API를 지원하면 (대부분의 모바일 브라우저) OS의 공유 시트가 떠서 카카오·문자·메일 등으로 보낼 수 있습니다.
- 지원하지 않으면 (대부분의 데스크톱 브라우저) `navigator.clipboard.writeText`로 링크를 클립보드에 넣고 "초대 링크를 복사했어요. 친구에게 보내주세요." 토스트를 띄웁니다.
- 사용자가 OS 시트에서 "취소"를 눌러도 클립보드 폴백으로 빠지지 않습니다 — 의도적인 취소를 존중.

### Q15. (가) 챌린지를 "성공"하려면 무엇을 해야 하나요?

규칙은 코드 한 줄로 정의돼 있습니다.

> **그 챌린지에서 내가 인증한 횟수**가 **주 목표 횟수** 이상이면 성공.

예를 들어 주 3회로 설정한 2주짜리 챌린지에서, 내가 총 3번 이상 인증했으면 성공입니다. 4번 5번 더 해도 그냥 성공이고, 2번 했으면 실패입니다.

⚠️ **"주(week) 목표 횟수"라는 이름이지만 실제로는 "챌린지 전체 기간 동안 누적 횟수"** 입니다. 7일짜리든 28일짜리든 같은 `goalCount` 횟수만 채우면 됩니다 — 코드상 별도 주(週) 단위 리셋 없음. 이 이름과 의미의 불일치는 PO와 한 번 정렬할 가치가 있는 영역입니다.

POC(개념 증명) 단계라서 실패해도 **실제로 돈이 빠져나가지는 않습니다**. 회고 화면에서 "예상 벌금"으로만 표시되고, 그 옆에 "표시 전용 · 실제 결제 없음"이라고 명시돼 있습니다.

### Q16. 벌금은 어떻게 계산되나요?

성공한 사람은 0원, 실패한 사람은 본인이 설정한 **벌금 금액 전액**입니다. 분할 계산은 없습니다.

홈 카드와 챌린지 상세에 보이는 "모인 예정 벌금"은 다음 식입니다.

```
모인 예정 벌금 = 챌린지 참가자 수 × 벌금 금액
```

이는 "전원이 실패했을 경우 최대로 모일 금액"입니다. 회고에서 보는 본인 벌금은 본인 성공/실패에 따라 0원 또는 벌금 금액 전액입니다.

예: 벌금 5,000원짜리 챌린지에서 4명 중 2명이 실패했다면 → 실패자 각자 5,000원, 실제 모인 돈은 10,000원. 홈 카드의 "모인 예정 벌금"은 20,000원으로 표시됨 (전원 실패 가정 최대치).

### Q19. (나) 인증을 시도했을 때 거부되는 경우는 어떤 것들이 있나요?

| 거부 사유 | 사용자에게 보이는 메시지 | 어디서 검사 |
|----------|-------------------------|-------------|

| 메모 100자 초과 | 클라이언트가 입력 단에서 100자로 자르므로 보통 발생 안 함 | 클라/서버 zod |
| 챌린지 참가자가 아님 | "현재 참여 중인 챌린지를 찾을 수 없어요." | 서버 액션 → DB RLS |
| 챌린지가 active가 아님 또는 기간 밖 | "지금은 인증할 수 있는 기간이 아니에요." | 서버 액션 → DB RLS |
| 사진 5MB 초과 | "사진은 5MB 이하만 올릴 수 있어요." | 클라이언트 사전 |
| 사진 형식 지원 안 함 | "지원하지 않는 이미지 형식이에요." | 클라이언트 사전 |
| 빈 파일 (size < 1) | DB가 거부 (실제로는 클라가 사진 없음으로 처리) | 서버 storage |
| 로그인 만료 | (자동으로 `/login`으로 이동) | 미들웨어/withUser |

키워드는 운동 종류별 정해진 풀에서만 고를 수 있고, **1개~3개 필수**입니다. 0개로는 인증 불가, 4개 이상도 불가. 운동 종류별 풀 크기는 모두 12개씩이고, 한 번에 6~9개가 무작위로 표시됩니다.

사진은 **선택 사항**입니다. 사진 없이 인증해도 됩니다. 그 경우 토스트가 "사진 없이 인증됐어요"로 살짝 다르게 뜹니다.

**3중 방어 구조**:

1. **클라이언트 단** (브라우저 내 zod) — 빠른 피드백용
2. **서버 액션** (서버 함수) — 권한·시간 검사 + UX 메시지 분기
3. **DB 권한 규칙(RLS)** — 마지막 방어선. 화면을 어떻게 우회해도 DB가 거부

이렇게 같은 검증을 3번 하는 이유는, 어느 한 층이 우회되더라도 최종 데이터 무결성을 보장하기 위해서입니다.

## 5. 사진 업로드와 표시 — 어떻게 보안되나?

### Q25. 사진은 어떤 형식·크기까지 가능한가요?

|------|------|
| 파일 크기 | 최대 5MB (5,242,880 바이트) |

### Q26. 사진은 누구에게 어떻게 보이나요?

사진은 **비공개(private) Storage 버킷**에 저장됩니다. 직접 URL로는 접근 불가능하고, 매번 **10분짜리 서명된 URL(signed URL)** 을 받아서 표시합니다.

사진 경로(파일 위치)는 다음 형식입니다.

```
{업로더_userId}/{challengeId}/{actionLogId}-{무작위문자열}.{확장자}
```

이 경로 구조에 따라 다음과 같이 권한이 강제됩니다.

| 동작 | 누가 가능 |
|------|----------|
| 사진 업로드 | 본인 폴더에만 (`auth.uid() = path 첫 세그먼트`) |
| 사진 보기 | 그 챌린지의 그룹 멤버 (path 두 번째 세그먼트로 챌린지 → 그룹 멤버십 확인) |
| 사진 삭제 | 본인 폴더만 |
| 사진 수정 | 누구도 불가 (UPDATE 정책 없음, 사진 객체는 immutable) |

업로드 자체는 2단계로 진행됩니다.

1. 인증 행을 먼저 DB에 만듦 (사진 없이) → `actionLogId` 확보
2. 그 ID를 path에 넣어 Storage 업로드 → `update_action_log_photo_path` RPC로 path 컬럼 갱신

업로드 실패 시 인증 행은 그대로 살아남고, 사용자에게는 "사진 없이 인증됐어요" 토스트가 뜹니다 (비파괴 폴백).

## 6. 시간 기준 — "당일 인증"이 정확히 뭔가?

이 질문에 대한 답은 **약간 의외**입니다. 코드를 보니 "당일"이라는 개념이 인증 자체에는 없고, 알림에만 있었습니다.

### Q28. 챌린지 인증은 시간상 어디부터 어디까지 가능한가요?

**챌린지의 시작 시각부터 종료 시각까지** 입니다. 예를 들어 7일짜리 챌린지가 5월 10일 오후 3시에 시작됐다면, 5월 17일 오후 3시까지 인증할 수 있습니다.

여기서 "오늘 자정~다음 자정" 같은 일별 경계는 없습니다. **챌린지 시작 순간부터 24시간 단위로 종료까지 연속된 한 덩어리의 시간 윈도우**입니다.

종료 시각이 지나면 DB 단에서 인증을 거부합니다. 챌린지의 status가 'closed'로 바뀌지 않아도 시간 윈도우가 지나면 그냥 막힙니다.

### Q29. 그러면 "하루에 한 번만 인증"이라는 제한이 있나요?

**없습니다**. 코드상 같은 챌린지에 하루에 여러 번 인증해도 막히지 않고, 그만큼 **인증 횟수가 누적**됩니다. 즉 주 목표가 3회인데 하루에 3번 인증하면 그날로 목표를 채울 수도 있습니다.

⚠️ 만약 "하루 1회"가 의도된 정책이라면 현재 코드에는 그 제한이 없습니다 — 별도 검토 필요 영역입니다. RLS 정책이나 UI 가드 어디에서도 "오늘 이미 인증했음" 체크가 없습니다.

### Q30. 그럼 "당일"이라는 개념이 어디 나오긴 하나요?

네, **"운동 시작 푸시 알림"** 한 곳에 있습니다.

홈이나 챌린지 화면에서 "운동 시작" 버튼을 누르면, 그룹원에게 "JJ님이 운동을 시작했어요!" 같은 푸시가 갑니다. 이 알림은 **하루에 한 번만** 보냅니다(스팸 방지). 같은 사람이 같은 챌린지에서 두 번 누르면, 두 번째부터는 "오늘 시작 알림은 이미 보냈어요" 토스트로 끝납니다.

여기서 "하루"의 기준은 **한국 시간(KST) 기준 자정**입니다. 자정이 지나면 다시 한 번 보낼 수 있습니다.

```
"오늘"의 시작 = 한국 시간 자정 (UTC 기준 전날 15시)
```

## 7. 푸시 알림 — 정확한 동작

### Q32. 푸시 알림은 종류가 몇 가지인가요?

3가지입니다.

| 알림 종류 | 누구에게 | 언제 | 본문 |
|----------|---------|------|------|
| **시작 알림** (`start`) | 그룹 멤버 전원 | 마지막 사람이 서약서에 서명해서 챌린지가 막 시작된 순간 | 제목: "챌린지 시작이에요" / 본문: "모두 서명했어요. 오늘부터 시작!" |
| **운동 시작 알림** (`start` 카테고리 안) | 그룹 멤버 (보낸 사람 제외) | 누군가 "운동 시작" 버튼을 누른 순간 (하루 1회) | 제목: "운동 시작" / 본문: "○○님이 운동을 시작했어요!" |
| **마감 임박 알림** (`deadline`) | 그룹 멤버 전원 | 챌린지 종료 12~36시간 전 (정기 작업) | 제목: "마감 24시간 전" / 본문: "아직 못 한 날이 있다면 지금!" |

알림 종류는 사용자 설정에서 두 가지 카테고리로 토글할 수 있습니다.

| 설정 토글 | 영향 |
|----------|------|
| **시작 알림** ON/OFF | "시작 알림" + "운동 시작 알림" 같이 켜고 꺼짐 (둘 다 `start` 카테고리) |
| **마감 임박 알림** ON/OFF | "마감 임박 알림"만 |

기본값은 둘 다 켜짐. 둘 다 끄면 푸시 구독 자체가 자동으로 해지됩니다(서버에서 user_id 기준으로 row 전체 삭제).

## 8. 참여 전/후 상태 — 합류 시점에 따라 무엇이 달라지나요?

이 부분이 질문 중 **가장 미묘한 곳**입니다. 코드 동작을 차근차근 정리합니다.

### Q37. 챌린지에는 어떤 상태들이 있나요?

4단계입니다.

| 상태 | 한국어 표현 | 어떻게 진입하는가 |
|------|------------|------------------|
| `pending` | 대기 중 (아직 멤버가 다 서명 안 함) | 챌린지를 처음 만든 직후 |
| `accepted` | 수락됨 | 코드상 명시적으로 진입시키는 곳이 없음. RLS의 update 정책 with_check에만 등장 — 실질적으로 사용 안 됨 |
| `active` | 진행 중 | 마지막 멤버가 서명한 순간, DB 함수가 자동으로 전이 |
| `closed` | 종료됨 | 코드상 자동 전이 트리거가 없음. 수동/향후 구현 영역 |

⚠️ **종료 시각(`end_at`)이 지나도 `status`가 자동으로 `closed`로 바뀌지는 않습니다.** 회고 화면은 이를 감안해서 "active이고 종료 시각이 지났음" 또는 "closed" 두 경우 모두를 "끝난 챌린지"로 처리합니다. 인증 차단도 status가 아니라 시간 윈도우로 합니다.

상태 전이 동시성:

- `sign_and_maybe_activate` 함수가 `for update`로 행 단위 락을 걸어 이중 전이를 방지합니다.
- `security definer`로 실행돼서, 마지막 서명자가 오너가 아니어도 status 전이 UPDATE가 통과합니다(0006번 마이그레이션의 명시적 수정).

- pending → active 전이 함수: `supabase/migrations/0006_rpc_activate_via_definer.sql:14-53`
- 동시성 락: `supabase/migrations/0006_rpc_activate_via_definer.sql:39-46` — `for update` + `c.status in ('pending','accepted')`

### Q38. ⭐ 핵심: 친구가 그룹에 새로 들어올 때, 진행 중 챌린지에 자동 참여하나요?

**그 그룹의 챌린지가 어느 상태냐에 따라 다릅니다**. 이게 질문하신 "참여 전 챌린지를 참여한 사람들만 진행, 참여 후에는 참여한 사람들 포함 이어서 진행"의 정확한 코드 구현입니다.

| 친구가 들어온 시점 | 결과 |
|-------------------|------|
| 그룹에 **챌린지가 아직 없음** | 그룹원으로만 추가됨. 다음 챌린지가 만들어지길 기다림 |
| 그룹에 **`pending` 챌린지가 있음** (=서약서가 만들어졌고 아직 모두 서명 전) | 그룹원으로 추가 + **자동으로 그 챌린지의 참가자가 됨**. 서약서에 본인 이름이 추가되고 서명할 수 있음 |
| 그룹에 **`active` 챌린지가 있음** (=이미 진행 중) | 그룹원으로만 추가됨. **진행 중 챌린지에는 참여하지 못함** ("freeze") |
| 그룹에 **`closed` 챌린지만 있음** | 그룹원으로만 추가됨 |

이 분기는 `accept_invite` RPC 안에 명시적으로 구현돼 있고, 코드 주석에 의도가 적혀 있습니다.

> pending 챌린지가 있으면 참가자로 편입 (active 이후는 freeze — PRD AC-6).

### Q39. 그러면 "진행 중 챌린지에 늦게 들어온 사람"은 무엇을 보게 되나요?

다음과 같이 보입니다.

- **홈 화면**: 그 그룹 카드는 보입니다. 하지만 진행 중 챌린지의 멤버 목록과 진행률에는 본인이 없습니다.
- **서약서 화면(`/pledge`)**: "아직 서명할 서약서가 없어요" 빈 화면이 뜹니다. 진행 중 챌린지는 이미 서명이 끝났고, 코드상 pending/accepted만 조회하기 때문입니다.
- **인증 화면(`/action`)**: 본인이 참가자가 아니므로 챌린지를 못 찾고 "현재 활성 상태인 챌린지가 없어요" 안내가 뜹니다.
- **피드 화면(`/feed`)**: 비슷하게 빈 상태가 됩니다.
- **챌린지 상세 화면**: URL을 직접 알아서 들어가면 멤버 진행률·피드 등은 볼 수 있지만, "운동 시작" 버튼은 안 뜹니다 (참가자가 아니므로).
- **푸시 알림**: 알림 발송 시 `challenge_participants` 기준으로 수신자를 골라서 보내므로, 늦게 들어온 사람은 그 진행 중 챌린지의 알림을 받지 않습니다.

즉, 친구가 챌린지가 시작된 후에 들어왔다면 **그 챌린지가 끝날 때까지는 사실상 구경꾼**이고, 다음 챌린지를 기다려야 합니다.

### Q40. ⚠️ 그룹 오너 자신은 챌린지에 어떻게 참가하나요?

**여기에 코드상 갭(gap)이 보입니다.**

`challenge_participants` 테이블에 INSERT하는 코드 경로는 **딱 한 곳뿐**입니다 — `accept_invite` 함수.

그런데:
- 그룹을 만들면 오너는 `group_members` 테이블에는 자동 들어갑니다 (역할: owner). `create_group_with_owner` RPC가 한 트랜잭션에서 처리.
- 챌린지를 만들면 `challenges` 테이블에 row만 생기고, 참가자 정보는 자동으로 안 들어갑니다 (`createChallenge` 액션은 `challenges` insert만).
- 즉 **오너 본인은 자기가 만든 챌린지의 참가자로 자동 등록되지 않습니다**.

이게 의도된 동작이라면, 오너도 본인의 초대 링크를 본인이 클릭해서 `accept_invite`를 통과시켜야 자기 챌린지의 참가자가 됩니다. 하지만 그게 일반적인 흐름은 아니라, **운영상 확인이 필요한 갭**으로 보입니다.

가능성:
- (a) 코드에서 누락됐고, 실제로는 어딘가의 운영 도구나 시드 스크립트가 채워주고 있을 수 있음
- (b) 의도적이고, 오너도 자기 초대 링크로 한 번 들어와야 함 — 단, 그 경우 `accept_invite`의 "이미 멤버" 분기로 빠져서 challenge_participants에 추가가 안 될 가능성 (Q41 참고)
- (c) 아직 구현 중이거나 향후 트리거로 추가 예정

⚠️ **이 부분은 PO나 백엔드 담당자에게 한 번 확인이 필요한 영역입니다.** 실제 운영 환경에서 오너 1명이 그룹/챌린지 만들고 바로 인증을 시도해 봤을 때 어떤 일이 벌어지는지 검증이 필요합니다.

### Q41. 다음 챌린지가 만들어지면 기존 그룹원들이 자동 합류하나요?

**그것도 자동으로 되지 않는 것으로 보입니다**. 같은 이유입니다.

`createChallenge`(새 챌린지를 만드는 함수)는 `challenges` 테이블에만 row를 추가합니다. 기존 그룹원을 새 챌린지에 자동으로 묶어주는 코드가 없습니다.

이론적으로는 기존 멤버도 한 번 더 초대 링크를 거치면 새 챌린지에 들어갈 텐데, **이미 그룹 멤버인 사람은 `accept_invite`에서 "이미 멤버" 분기로 빠지고**, 그러면 `challenge_participants`에도 추가되지 않습니다 (코드의 `if not v_already_member then` 블록 안에서만 INSERT가 일어남).

### Q42. 서약서 서명은 어떻게 진행되나요?

서약서 화면(`/pledge`)에 들어가면, 내가 참가 중인 pending/accepted 챌린지 1개가 자동으로 표시됩니다. 거기에:

- 챌린지 정보 (제목·기간·주 목표·예정 벌금)
- 멤버별 서명 현황 ("서명 완료" 또는 "대기 중")
- 본인 이름이 들어간 동의 문구 + 체크박스 + "서명하고 참여" 버튼

이 보입니다. 본인이 이미 서명했으면 "이미 서명했어요" 메시지가 보입니다.

동의 문구 예시 (본인 이름이 자동 삽입):

```
나 [이름]은(는) 위 조건에 동의합니다. 어긴 경우 공동 통장에 입금할게요.
```

내가 마지막 서명자라면, 서명 버튼을 누르는 순간 두 가지 일이 일어납니다.

1. 챌린지가 `active` 상태로 바뀌고, `start_at`이 현재 시각으로, `end_at`이 시작 + 기간 일수로 자동 계산됨
2. 시작 알림이 그룹원 전체에게 자동으로 발송됨 (실패해도 서명 자체는 성공으로 끝남 — fire-and-forget)

토스트도 다르게 뜹니다.

| 상황 | 토스트 |
|------|------|
| 일반 서명 (아직 다른 사람 미서명) | "서명했어요!" |
| 마지막 서명 (전원 완료) | "전원 서명 완료! 챌린지가 시작됐어요." |

## 9. 권한 — 그룹장(생성자)과 멤버(참여자)의 차이

권한은 두 단계로 강제됩니다.

1. **DB 권한 규칙(RLS)** : 마지막 진실. 화면에서 어떻게 우회해도 DB가 거부합니다.
2. **화면 분기** : 편의 (멤버에게는 오너 전용 버튼이 아예 안 보이게)

### Q43. 그룹장(오너)만 할 수 있는 일은?

| 동작 | 비고 |
|------|------|
| 그룹 만들기 | 자기 자신을 owner로 등록 |
| 그룹 정보 수정 | 그룹 이름 등 |
| 새 서약서(챌린지) 만들기 | 멤버는 만들 수 없음 |
| 서약서 수정 | 단, **아직 멤버가 다 서명하지 않은 시점(`pending`)까지만** 가능. 한 번 active가 되면 오너도 못 고침 |
| 친구 초대 링크 만들기 | 멤버는 초대 못 함 |
| 초대 링크 목록 조회 / 삭제 | |
| 멤버 강제 탈퇴 | 본인이 본인을 빼는 건 멤버도 가능 |
| 정산 계좌 등록 | 그룹 만들 때 같이 입력 (선택) |

### Q44. 멤버(참여자)도 할 수 있는 일은?

| 동작 | 비고 |
|------|------|
| 그룹·챌린지 정보 보기 | 비멤버는 DB가 차단해서 조회 자체가 안 됨 |
| 다른 멤버의 정보 보기 (display_name 등) | 같은 그룹에 속한 사람만 — DB 정책 `users_select_self_or_group`이 두 단계 그룹원 교집합으로 판정 |
| 서약서에 서명 | 자기 자신만 (`signed_at` 컬럼 update) |
| 인증 작성 | 챌린지가 진행 중이고 기간 안일 때만 |
| 인증 5분 내 수정 | 단, AI 일기 컬럼은 트리거가 잠금 |
| 다른 멤버의 인증에 응원(🔥💪👏) | **자기 인증에는 응원 불가** |
| 운동 시작 알림 보내기 | 하루 1회 |
| 자기 그룹 탈퇴 | |
| 푸시 알림 설정 | 본인 것만 |
| **정산 계좌 평문 보기** | 그룹 멤버이기만 하면 누구나 볼 수 있음. 오너 권한이 아니어도 됨 |

### Q45. 응원(Kudos) 규칙을 더 자세히 알려주세요.

3종 이모지: **🔥 / 💪 / 👏** 입니다. 문화적 의미는 코드상 정의되어 있지 않습니다.

규칙:

| 규칙 | 동작 |
|------|------|
| 같은 인증에 같은 이모지를 두 번 누르면 | 토글됨 (취소). DB가 `unique(action_log_id, user_id, emoji)` 제약으로 강제 |
| 같은 인증에 다른 이모지 추가 | 가능 — 한 사람이 한 인증에 🔥💪👏 모두 가능 |
| 자기 인증에 응원 | **불가** — DB와 UI 양쪽에서 차단 |
| 비멤버가 응원 | **불가** — DB가 차단 |
| 응원이 즉시 화면에 반영 | 낙관적 업데이트 (낙관적으로 +1 했다가 실패 시 토스트로 알림) |

### Q46. 누구도 직접 할 수 없는 일은?

DB 차원에서 모든 사용자를 막고, 정해진 안전한 함수로만 통과시키는 영역입니다.

| 동작 | 차단 이유 |
|------|----------|
| 그룹 멤버 직접 추가 | 반드시 `accept_invite` 함수를 거쳐야 함 (인원 제한·만료 검증 한 번에) |
| 챌린지 참가자 직접 추가 | 같은 이유 |
| AI가 만든 일기 요약 직접 수정 | DB 트리거가 막음 (위변조 방지) |
| 사진 객체 수정 (UPDATE) | Storage 정책에 UPDATE 없음 — 사진은 immutable |
| `events` 표 조회 (분석 데이터) | service_role만 가능 |
| `ai_cost_log` 표 조회/수정 | service_role만 |
| 진행 중(`active`) 챌린지 정보 수정 | 오너도 불가 — 약속을 지키는 의미. DB 정책의 `using` 절이 `status = 'pending'`만 허용 |

## 부록 A. 용어 풀이 (가나다순)

본문에서 짧게 풀어 쓴 단어를 한 번 더 정리합니다.

- **AC** : Acceptance Criteria. 인수 기준. 기획서에서 "여기까지 되면 완성"의 정의.
- **active** : 챌린지 4단계 중 "진행 중" 상태.
- **AES-256-GCM** : 정산 계좌번호 암호화에 쓰는 표준 대칭 암호. DB는 평문을 모름.
- **closed** : 챌린지 4단계 중 "종료" 상태. 코드상 자동 전이는 없음.
- **idempotency** : 같은 호출을 여러 번 해도 결과가 한 번 호출한 것과 같음. 중복 발송 방지에 사용.
- **immutable** : 한 번 만들어진 후엔 수정 불가. 사진 객체와 AI 일기가 해당.
- **인증(action_log)** : 운동을 했다는 기록. 사진 + 키워드 + (선택)메모로 구성.
- **KST** : 한국 표준시. UTC+9. 새벽 2-7시 알림 보류, 운동 시작 알림의 "하루"가 모두 KST 기준.
- **낙관적 업데이트** : 서버 응답 전에 화면을 먼저 바꾸고, 실패 시 되돌리기. 응원 토글에 사용.
- **OTP** : One-Time Password. 일회용 비밀번호. 여기선 매직링크 인증 방식.
- **오너(owner)** : 그룹을 만든 사람. 그룹당 1명. 권한이 가장 강함.
- **참가자(participant)** : 챌린지 단위로 합류한 사람. 그룹 멤버라고 자동으로 참가자가 되는 건 아님 (Q40 참고).
- **pending** : 챌린지 4단계 중 "서명 대기" 상태.
- **POC** : Proof of Concept. 제품화 전 핵심 가설 검증 단계. 이 앱은 POC라서 실제 결제는 안 함.
- **PWA** : Progressive Web App. 휴대폰 홈 화면에 설치할 수 있는 웹 앱.
- **Quiet Hours** : 푸시 발송이 보류되는 시간대. 이 코드에서는 KST 02:00~07:00.
- **RLS** : Row Level Security. 행 단위 접근 권한을 DB가 직접 강제.
- **RPC** : Remote Procedure Call. Supabase의 Postgres 함수 호출.
- **service_role** : 서버에서만 쓰이는 슈퍼 권한 키. 일반 사용자 키와 구분.
- **signed URL** : 서명된 임시 URL. 비공개 사진을 잠깐 보여줄 때 사용 (10분 유효).
- **토스트(toast)** : 화면에 잠깐 떴다 사라지는 메시지. 모달과 달리 사용자 행동을 막지 않음.
- **VAPID** : Voluntary Application Server Identification. Web Push 발신자 인증용 키 쌍.
- **Web Share API** : 브라우저가 OS의 공유 시트를 띄우게 하는 표준 API.

## 부록 B. 이 문서가 다루지 않은 영역 / 추가 확인 필요

**의도적으로 제외**

- AI 일기 비용/예산 세부 (`src/lib/ai/cost.ts`) — 운영자만 신경쓰는 영역
- 분석 이벤트 스키마 (`src/lib/analytics/schema.ts`) — 백엔드 분석 파이프라인 영역
- 카카오 로그인 — 비활성화 상태
- middleware의 정확한 쿠키 동기화 패턴 — Supabase ssr 패턴 그대로

**⚠️ 운영 확인이 필요한 영역**

- **그룹 오너의 첫 챌린지 자동 합류** (Q40) — 코드만 봐서는 오너가 자동으로 challenge_participants에 들어가지 않음. 실제 운영 환경에서 검증 필요.
- **기존 그룹원의 다음 챌린지 자동 합류** (Q41) — 같은 맥락. 실제 동작 검증 필요.
- **챌린지 자동 종료(closed 전이)** (Q37) — 코드상 트리거 없음. status가 영원히 active로 남는지 운영 확인 필요.
- **"하루 1회 인증" 정책** (Q29) — 만약 의도된 정책이라면 현재 구현에 빠져 있음.
- **goalCount의 의미** (Q15) — 이름은 "주 목표 횟수"인데 동작은 "전체 기간 누적 횟수". PO와 정렬 가치 있음.
- **새벽 시간대 알림 재시도 부재** (Q33) — 새벽에 발생한 알림 이벤트는 그대로 보류로 끝남. 의도적인지 확인 필요.
