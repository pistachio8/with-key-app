# docs/junior/CLAUDE.md — Claude Code · Cursor 작업 규칙 (주니어판)

> **원본**: [`../CLAUDE.md`](../CLAUDE.md). 이 문서는 같은 규칙을 풀어 쓴 버전입니다. 두 문서가 어긋나면 원본이 기준입니다.

## 이 문서는 무엇인가

`with-key` 저장소에서 **AI 코딩 도구(Claude Code · Cursor)** 가 코드를 쓸 때 지켜야 할 행동 지침입니다. 안드레아 카파시(Andrej Karpathy)가 공개한 [CLAUDE.md 스킬 문서](https://github.com/forrestchang/andrej-karpathy-skills/blob/main/CLAUDE.md)를 기반으로, 이 프로젝트 맥락에 맞게 다듬었습니다.

**핵심은 LLM의 편향을 누르는 것입니다.** LLM은 기본적으로 "뭔가를 더 만드는" 방향으로 기울어 있습니다. 요청받지 않은 기능·추상화·유연성을 덧붙이려 하고, 주변 코드까지 "개선"하려 듭니다. 이 문서의 4가지 규칙은 그 편향을 눌러, **요청받은 것만 정확히 만들게** 하기 위한 것입니다.

---

## 트레이드오프: 이 규칙이 희생하는 것

이 가이드는 **속도보다 신중함**을 우선합니다. 작은 변경 하나에도 "먼저 가정을 밝히고 → 선택지를 제시하고 → 확인받고" 같은 단계를 요구합니다. 그래서 **단순한 작업**(오타 수정·문구 변경·한 줄 버그 픽스)에서는 이 절차를 전부 지키면 오히려 비효율적입니다.

신중함을 요구하는 대상은 **"되돌리기 비용이 큰" 변경**입니다. 예:

- 마이그레이션(`supabase/migrations/*.sql`) 추가·수정
- `middleware.ts`의 인증 가드 로직 변경
- `src/lib/validators/`의 zod 스키마 시그니처 변경
- 여러 route에서 공유하는 `src/lib/**` API 변경

이런 영역은 규칙을 전부 지키고, 그렇지 않은 영역은 **상황에 맞게** 판단합니다.

## 1. 코드 작성 전에 생각하기

> 추측하지 말고, 혼란을 숨기지 말고, 선택지를 드러내라.

구현을 시작하기 **전에** 반드시 다음 네 가지를 수행합니다. 구현 후에 하면 의미가 없습니다 — 이미 만들어진 코드를 보면 "이게 맞겠지"로 합리화하게 됩니다.

### 1-1. 가정을 명확히 밝힌다

어떤 입력이 들어올지, 어떤 경계 조건이 있는지, 어떤 환경에서 동작할지를 **말로 적습니다.** 불확실하면 구현하지 말고 질문합니다.

- 나쁜 예: "로그인 상태 판단 로직을 추가했습니다." (어떤 기준으로? 쿠키? 세션?)
- 좋은 예: "Supabase 세션 쿠키가 있고 만료되지 않았을 때만 `authenticated=true`로 판정한다고 가정하고 구현하겠습니다. 다른 가정이 있으면 알려주세요."

### 1-2. 여러 해석이 가능하면 모두 제시한다

요청이 모호할 때 **조용히 하나를 골라 구현하지 않습니다.** 해석 A·B·C를 나열하고, 각각의 장단점을 짧게 적고, 권장안을 고릅니다.

### 1-3. 더 단순한 접근이 있다면 반드시 언급한다

사용자가 요청한 방향이 최선이 아닐 수 있습니다. 더 단순한 대안이 보이면 **언급합니다.** 필요하면 사용자 의견에 **반박해도 됩니다** (이건 서비스가 아니라 협업입니다).

### 1-4. 명확하지 않으면 멈춘다

"대충 이렇게 하면 되겠지"로 시작하지 않습니다. 무엇이 헷갈리는지 **구체적으로** 말하고 질문합니다. "뭔가 잘 모르겠다"가 아니라 "이 입력이 `null`일 때의 동작이 스펙에 없습니다 — (a) 빈 배열 반환 (b) 에러 throw 중 어느 쪽이어야 할까요?" 수준이어야 합니다.

## 2. 단순함 우선

> 문제를 해결하는 최소한의 코드만 작성하라. 불필요한 것은 넣지 마라.

LLM이 가장 자주 저지르는 실수는 **"보험용 코드"를 추가하는 것**입니다. "혹시 모르니까", "나중에 확장 가능하게". 이 프로젝트에서는 금지입니다. POC 기간 2주 안에 **검증할 단 하나의 질문**에 집중하기 위해서입니다.

다음 다섯 가지는 명시적으로 하지 않습니다.

- **요청받지 않은 기능 추가 금지.** 요구사항에 없는 "좋아 보이는 기능"을 덧붙이지 않습니다.
- **한 번만 쓰이는 코드에 대한 추상화 금지.** 호출자가 1명이면 함수·클래스·훅으로 감싸지 않습니다. 두 번째 호출자가 나타날 때 추상화해도 늦지 않습니다.
- **요청되지 않은 "유연성"·"설정 가능성" 추가 금지.** 파라미터 5개 중 4개가 항상 기본값이면 그 4개는 빼고 만듭니다.
- **발생할 수 없는 상황에 대한 에러 처리 금지.** 타입 시스템이 보장하는 조건을 런타임에서 또 검사하지 않습니다. zod로 이미 검증된 값을 다시 `if (value == null)` 체크하지 않습니다.
- **200줄로 작성했는데 50줄로 가능하다면 다시 작성합니다.**

### 스스로 점검하는 질문

> "시니어 엔지니어가 이걸 보고 **'과하게 복잡하다'** 고 할까?"

그렇다면 단순화합니다. 이 질문을 PR 제출 전에 한 번, 머지 직전에 한 번 던지는 것이 가장 효과적입니다.

## 3. 외과적 수정 (Surgical edits)

> 필요한 부분만 건드려라. 정리는 내가 만든 것만 한다.

### 3-1. 기존 코드를 수정할 때의 원칙

- **주변 코드·주석·포맷을 "개선"하지 않습니다.** 같은 파일을 열었다고 해서 눈에 거슬리는 부분을 함께 고치지 않습니다. 변경 범위가 넓어질수록 리뷰 비용과 회귀 가능성이 모두 커집니다.
- **문제 없는 코드를 리팩토링하지 않습니다.** "더 깔끔하게 쓸 수 있을 것 같아서"는 리팩토링 사유가 아닙니다. 별도 요청/PR로 분리합니다.
- **기존 스타일을 따릅니다.** 내 방식이 더 좋아 보여도 같은 저장소 안에서는 기존 스타일이 정답입니다. 스타일을 바꾸고 싶으면 팀 논의를 거쳐 코드베이스 전체를 한 번에 정리합니다.
- **관련 없는 죽은 코드(dead code)를 발견하면 언급만 하고 삭제하지 않습니다.** 언급은 PR 설명의 "후속 작업" 섹션으로 남깁니다.

### 3-2. 변경으로 생긴 불필요한 코드 처리

- **내 변경으로 인해** 사용되지 않게 된 import·변수·함수만 제거합니다.
- **기존에 있던** 죽은 코드는 요청이 없으면 건드리지 않습니다.

### 3-3. 검증 기준

**변경된 모든 라인은 사용자의 요청과 직접 연결되어야 합니다.** 어떤 라인의 변경 이유를 한 줄로 설명할 수 없다면, 그 라인은 되돌립니다. PR diff를 열어 "이 라인은 왜 바뀌었지?"에 모두 답할 수 있어야 합니다.

## 4. 목표 중심 실행

> 성공 기준을 정의하고, 검증될 때까지 반복하라.

작업을 **"검증 가능한 목표"** 로 바꿔 적습니다. 모호한 요청은 그대로 받지 않고, 먼저 검증 가능한 형태로 번역합니다.

| 모호한 요청 | 검증 가능한 목표 |
|---|---|
| "validation 추가" | 잘못된 입력을 막는 테스트 작성 → 통과시키기 |
| "버그 수정" | 버그 재현 테스트 작성 → 통과시키기 |
| "X 리팩토링" | 변경 전/후 테스트 모두 통과 확인 |

여러 단계 작업이면 간단한 계획을 적습니다. 각 단계마다 **"어떻게 통과를 확인할지"** 를 같이 적는 것이 핵심입니다.

```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

### 왜 검증 방법까지 적어야 하나

강한 성공 기준은 LLM/사람이 스스로 반복 실행할 수 있게 해줍니다. "테스트 명령어를 실행해 다 통과하는지 본다"는 명확합니다. 반면 약한 기준("잘 되게 만들기")은 매번 사용자에게 "이거 맞나요?"를 되묻게 됩니다. 약한 기준으로 시작하면 작업이 끝나지 않습니다.

## 용어집

- **AC (Acceptance Criteria, 수락 기준)** — 기능이 "완료"로 간주되기 위한 체크 조건. 이 문서에서는 "성공 기준"과 같은 맥락.
- **AI 코딩 도구** — Claude Code, Cursor, GitHub Copilot처럼 코드베이스를 읽고 제안·수정까지 수행하는 도구. 이 저장소는 Claude Code와 Cursor 사용을 전제로 합니다.
- **Andrej Karpathy** — Tesla AI 책임자·OpenAI 창립 멤버였던 엔지니어/연구자. "Software 2.0", "nanoGPT" 등으로 알려져 있으며, LLM을 활용한 코딩 워크플로우에 대한 공개 가이드를 다수 작성.
- **Dead code (죽은 코드)** — 호출되지 않거나, 조건상 실행될 수 없는 코드. 리팩토링 대상으로 보이기 쉽지만, 이 문서 맥락에서는 "요청 범위 밖"으로 취급해 건드리지 않고 언급만 합니다.
- **LLM (Large Language Model, 대규모 언어 모델)** — Claude, GPT 계열처럼 대규모 텍스트로 학습된 생성형 AI 모델. 이 문서에서는 Claude Code·Cursor 뒤에서 동작하는 모델 일반을 가리킵니다.
- **POC (Proof of Concept, 개념 증명)** — 본격 개발에 들어가기 전, 핵심 가설이 실제로 작동하는지 짧은 기간 안에 확인하기 위한 최소 구현. `with-key`는 2주 POC.
- **RSC (React Server Component, 서버 컴포넌트)** — 서버에서 렌더링돼 HTML로 넘어오는 React 컴포넌트. 클라이언트 JS 번들에 포함되지 않아 크기·지연 면에서 유리.
- **Surgical edit (외과적 수정)** — 요청과 직접 연결된 라인만 건드리는 수정 방식. 의학의 최소 침습 수술에서 따온 비유로, 주변 코드의 정리·포맷·리팩토링을 섞지 않습니다.
- **zod** — TypeScript용 스키마 선언/검증 라이브러리. `with-key`에서는 `src/lib/validators/`에 두고, 런타임 입력 검증과 TypeScript 타입 도출(`z.infer`)을 동시에 수행합니다.
