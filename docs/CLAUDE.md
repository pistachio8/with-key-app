안드레아 카파시(andrej-karpathy-skills)의 문서를 참조합니다.(https://github.com/forrestchang/andrej-karpathy-skills/blob/main/CLAUDE.md)

LLM이 코딩할 때 자주 발생하는 실수를 줄이기 위한 행동 지침입니다. 필요에 따라 프로젝트별 규칙과 함께 사용하세요.

트레이드오프:
이 가이드는 속도보다 신중함을 우선합니다. 단순한 작업에서는 상황에 맞게 판단하세요.

1. 코드 작성 전에 생각하기

추측하지 말고, 혼란을 숨기지 말고, 선택지를 드러내라

구현 전에 반드시:

가정을 명확하게 밝힌다. 불확실하면 질문한다.
여러 해석이 가능하면 모두 제시한다 — 조용히 하나를 선택하지 않는다.
더 단순한 접근이 있다면 반드시 언급한다. 필요하면 반박한다.
명확하지 않으면 멈춘다. 무엇이 헷갈리는지 구체적으로 말하고 질문한다. 2. 단순함 우선

문제를 해결하는 최소한의 코드만 작성하라. 불필요한 것은 넣지 마라

요청받지 않은 기능 추가 금지
한 번만 쓰이는 코드에 대한 추상화 금지
요청되지 않은 “유연성”, “설정 가능성” 추가 금지
발생할 수 없는 상황에 대한 에러 처리 금지
200줄로 작성했는데 50줄로 가능하다면 다시 작성하라

스스로에게 물어라:
"시니어 엔지니어가 이걸 보고 과하게 복잡하다고 할까?"
그렇다면 단순화하라.

3. 외과적 수정

필요한 부분만 건드려라. 정리는 내가 만든 것만 한다

기존 코드를 수정할 때:

주변 코드, 주석, 포맷을 “개선”하지 마라
문제 없는 코드를 리팩토링하지 마라
기존 스타일을 따른다 (내 방식이 더 좋아 보여도)
관련 없는 죽은 코드(dead code)를 발견하면 언급만 하고 삭제하지 마라

변경으로 인해 생긴 불필요한 코드 처리:

내 변경으로 인해 사용되지 않게 된 import/변수/함수만 제거
기존에 있던 죽은 코드는 요청 없으면 건드리지 않는다

검증 기준:
변경된 모든 라인은 사용자의 요청과 직접적으로 연결되어야 한다

4. 목표 중심 실행

성공 기준을 정의하고, 검증될 때까지 반복하라

작업을 검증 가능한 목표로 바꿔라:

"validation 추가" → "잘못된 입력 테스트 작성 → 통과시키기"
"버그 수정" → "버그 재현 테스트 작성 → 통과시키기"
"X 리팩토링" → "변경 전/후 테스트 모두 통과 확인"

여러 단계 작업이라면 간단한 계획을 작성:

1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]

강한 성공 기준은 스스로 반복 실행을 가능하게 한다.
약한 기준(예: “잘 되게 만들기”)은 계속해서 추가 설명을 필요로 한다.
