## 표기 규칙

- **Confidence**: 이 가정이 **참이라고 믿을 근거**의 강도. Low일수록 위험.
- **🪂 LoF (Leap of Faith)**: 틀리면 P1+P2 전체가 무너지는 핵심 가정. 가장 먼저 검증해야 함.

## 1. Value — 진짜 가치를 만드는가

| ID  | 가정                                                                                                                                              | 무엇이 잘못될 수 있나                                           | Confidence | 검증 방법                                                   | LoF |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------- | ---------- | ----------------------------------------------------------- | --- |
| V1  | 유저가 **현금화 불가 포인트**를 현금 벌금만큼의 손실회피 동기로 느낀다                                                                            | "어차피 현금도 아닌데" → 압박 약화 → 루프 동력 상실             | **Low**    | dogfood 그룹에 가상 잔액 차감 fake-door, 이탈/완료율 측정   | 🪂  |
| V2  | 그룹장이 종료 시 **"정산" 버튼을 실제로 누른다** (수동 트리거 의존)                                                                               | 안 누르면 보상이 멈춰 다음 챌린지 동기 붕괴                     | Med        | 종료 프롬프트 클릭률, 미클릭 시 auto-settle fallback 필요성 |     |
| V3  | (**공동 주머니=다음 챌린지 이월 확정**) 달성자 원금 환급 + 미달분이 **그룹의 다음 챌린지 보증금으로 이월**되는 방식을 공정·납득 가능하다고 느낀다 | 이월이라 손실감 약화 가능(그룹이 결국 다시 씀) → V1과 함께 검증 | **Low**    | 분배 규칙 컨셉 테스트(인터뷰)                               | 🪂  |
| V4  | 유저가 **자동검증 판정 결과를 신뢰·수용**한다                                                                                                     | 기계 판정 불신 → 이의제기 폭증 → 그룹장 부담 회귀               | Low~Med    | 자동판정 vs 그룹장판정 인식 A/B                             |     |

## 2. Usability — 쓸 줄 아는가

| ID  | 가정                                                    | 무엇이 잘못될 수 있나                           | Confidence | 검증 방법                                 | LoF |
| --- | ------------------------------------------------------- | ----------------------------------------------- | ---------- | ----------------------------------------- | --- |
| U1  | 유저가 **포인트 풀·분배 규칙을 이해**한다               | closed-loop 개념 복잡 → 정산 화면에서 이탈/혼란 | Med        | 첫 정산 화면 사용성 테스트(이해도 task)   |     |
| U2  | 오판 시 **이의제기·재인증 경로**를 쉽게 찾는다          | 막다른 길 → 억울함 → 신뢰 붕괴                  | Med        | 오판 시나리오 사용성 테스트               |     |
| U3  | **네이티브 즉석 촬영 강제**가 인증 마찰을 키우지 않는다 | 갤러리 차단 불편 호소 → 인증율 하락             | Med        | 촬영-only vs 갤러리 허용 인증 완료율 비교 |     |

## 3. Viability — 사업·법무가 받쳐주는가

| ID  | 가정                                                                | 무엇이 잘못될 수 있나             | Confidence | 검증 방법                                               | LoF |
| --- | ------------------------------------------------------------------- | --------------------------------- | ---------- | ------------------------------------------------------- | --- |
| Vi1 | 현금 충전 포인트의 **미사용분 환불 의무**(전자상거래법)를 감당 가능 | 환불 운영비·정책 미비 → 분쟁/제재 | **Low**    | 법무 검토 + 충전 없는 적립/번들 포인트(ⓑ안)로 우회 시작 | 🪂  |
| Vi2 | 일정 규모 초과 시 **선불전자지급수단** 등록 이슈를 회피 가능        | 규모 성장 시 라이선스 트리거      | Low        | 법무 검토, 발행 잔액 임계 모니터링                      |     |
| Vi3 | **원금 초과 분배가 사행성으로 분류되지 않는다**                     | 도박성 판정 → 서비스 중단 리스크  | **Low**    | 법무 검토, 분배를 "원금 환급 한도 내"로 설계            | 🪂  |
| Vi4 | **구독만으로 단위경제 성립** (개인 WTP 약함)                        | 개인 전환 저조 → 수익 공백        | Low        | 가격 민감도(Van Westendorp), B2B 파일럿                 |     |
| Vi5 | 0% rake로도 **구독이 운영비(AI·푸시·스토리지·정산)를 커버**         | 비용 > 구독 수익 → 적자 구조      | Low~Med    | 인증 1건당 비용 모델 × 구독 전환 가정                   |     |

## 4. Feasibility — 기술로 만들 수 있는가

| ID  | 가정                                                                                                          | 무엇이 잘못될 수 있나                                              | Confidence | 검증 방법                                                    | LoF |
| --- | ------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ | ---------- | ------------------------------------------------------------ | --- |
| F1  | (**부정탐지형 재정의**) 정상 사진을 **부정으로 오판하지 않으면서** 명백한 부정(중복·스크린샷·도용)을 잡아낸다 | 정상 사진 false-flag → 억울·신뢰 붕괴 / 부정 누락 → 포인트 farming | Low~Med    | 실사용 사진셋 false-flag rate 벤치("운동 분류"를 버려 위험↓) | 🪂  |
| F2  | 온디바이스 사전검증(Vision/ML Kit)이 iOS/Android에서 신뢰성 동작                                              | 기기 편차·실패 → UX 저하                                           | Med        | 실기기 PoC                                                   |     |
| F3  | anti-cheat(메타데이터·perceptual hash)이 우회 공격에 견고                                                     | 스크린샷·재촬영·메타 조작 우회 → 치팅                              | Low~Med    | red-team 테스트(우회 시나리오)                               |     |
| F4  | 포인트 원장(append-only+RPC)이 **동시성/정합성** 안전                                                         | 이중 정산·잔액 오류 → 금전 분쟁                                    | Med        | 트랜잭션 동시성 테스트, RLS 검증                             |     |
| F5  | 자동검증 비용이 0% rake + 구독 예산 안에 든다                                                                 | 호출 비용 폭증 → 적자                                              | Med        | 인증 1건당 비용 × 예상 볼륨                                  |     |

## 5. 우선순위화 — Riskiest Assumption 먼저 (impact × uncertainty)

> /pm-product-discovery:prioritize-assumptions 기준. **고임팩트 × 저confidence(=고불확실)** 가정이 leap-of-faith. 이걸 먼저 싸게 검증한다.

```
            높은 IMPACT
                │
   F1 정확도 🪂 │ Vi3 사행성 🪂
   V1 포인트동기🪂│ Vi1 환불의무 🪂
   V3 분배정서 🪂 │
 ───────────────┼─────────────── 높은 UNCERTAINTY →
   V4 자동판정신뢰 │ U1 이해도
   F3 anti-cheat │ Vi4/Vi5 단위경제
                │
            낮은 IMPACT
```

### 먼저 검증할 Top 5 (좌상단 = 가장 위험)

| 순위 | 가정                            | 영역        | 가장 싼 검증(다음 액션)                                                                                    |
| ---- | ------------------------------- | ----------- | ---------------------------------------------------------------------------------------------------------- |
| 1    | **F1 부정탐지 정밀도**          | Feasibility | 실사용 사진셋 false-flag rate 측정(정상→부정 오판, 코딩 PoC). 임계 미달이면 부정탐지 보수적 조정·사람검토↑ |
| 2    | **Vi1 + Vi3 규제(환불·사행성)** | Viability   | **법무 검토 1회** — 가장 싸게 GO/NO-GO 판가름. ⓑ안(적립/번들 포인트)으로 시작해 충전·초과분배는 통과 후    |
| 3    | **V1 포인트 손실회피 동기**     | Value       | dogfood 가상 잔액 차감 fake-door, 완료율 비교                                                              |
| 4    | **V3 분배 수용(공동 주머니)**   | Value       | 분배 규칙 컨셉 인터뷰 (공동 주머니 납득도)                                                                 |
| 5    | **V2 정산 트리거 실행**         | Value       | 종료 프롬프트 클릭률 + auto-settle fallback 설계 여부                                                      |

### 핵심 함의 (PRD로 넘기기 전 반영할 것)

1. **F1(정확도)와 Vi(규제)가 둘 다 leap-of-faith** — 둘 중 하나만 깨져도 P1+P2 페어가 무너진다. PRD 착수 전 **법무 검토 1회 + 정확도 PoC 1회**를 선행 조건으로 둘 것.
2. **규제 우회 설계**: startup-canvas ⓑ안(현금 충전 대신 적립·구독 번들 포인트)으로 MVP를 시작하면 Vi1·Vi3 위험이 ≈0으로 떨어진다. → **MVP는 ⓑ안으로 시작, 현금 충전은 법무 통과 후 토글.**
3. **정산 트리거 안전장치**: V2 대비 그룹장 수동 정산 + **마감 후 auto-settle fallback**(brainstorm E4)을 PRD에 함께 명세.
4. **정산 = 공동 주머니 방식 (확정 2026-06-02)**: 달성자 원금 환급 + 미달분은 **그룹 공동 주머니 → 다음 챌린지 보증금 이월**(현금화 불필요, 회식·기부 등 현금 용도 제외). 개인 재분배는 도박 위험으로 미채택. → V3·Vi3(사행성) 동시 완화.
